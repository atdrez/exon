Multipurpose Language

The Exon Language

A simple, composable language with no fixed domain. Configuration, scripting, document generation, automation, typed DSLs, one structured syntax, shaped to fit any problem.

Get Started Tutorial

Exon files define structured object graphs. Inheritance lets child files override any property from a base type without duplicating the rest. Bindings reference computed values by name across the same file. Components handle arithmetic, flow control, error handling, string manipulation, document transformations, or anything a component produces. Properties enforce type constraints, value ranges, and business rules at the field level through getter/setter pairs, self-validating data with no external schema language required. Add your own components in JavaScript and they slot in identically to the standard library.

Quick Look

A minimal Exon file is just an object literal. Add using fn.* to unlock built-in components.

using fn.*
using fn.process.env

{
    host: "localhost"
    port: coalesce { env { "PORT" } 8080 }
    debug: false
    version: 1.04
    validated: null
    tags: [ "dev", "internal" ]
    versionedTags: foreach {
        data: @root.tags
        do: string.join { parameter{} ":" @root.version }
    }
}

Running PORT=30 exon hello.exon prints the JSON representation:

        "dev",

    ]
}

Core Principles

Exon is built around a small number of foundational ideas. Understanding them explains the design decisions behind the language and the trade-offs that were deliberately made.

Simplicity

The language reduces to a single concept: a named object block with key-value properties. There are no special syntax forms for operators, loops, or functions. Every construct, from arithmetic to file I/O, is expressed as a named object call. This uniformity makes Exon fast to learn, predictable to read, and straightforward to extend without touching the grammar.

Extensibility

The language specification deliberately omits arithmetic, logic, flow control, and I/O. These capabilities are implemented as components (pluggable objects that the resolver dispatches to at runtime). A standard library ships with over 90 components covering math, strings, collections, serialization, and process interaction. You can add your own in JavaScript or TypeScript without modifying the runtime.

Easy to Port

The interpreter is a small tree-walking resolver with no bytecode compiler, no JIT, and no built-in type system. The entire pipeline is: lexer, parser, and resolver with a component registry. Porting Exon to a new host language requires implementing this pipeline over a map/array data model, nothing more.

Multipurpose

Because Exon has no built-in semantics beyond object resolution, the same syntax can power configuration management, document generation, build pipelines, GUI layout, CSS authoring, automation scripting, or any domain-specific language you need. Swap the component library and the language speaks a different domain, with no changes to the parser or resolver.

Type Agnostic

The language works over four primitive constructs (strings, numbers, booleans, and null) plus arrays and generic objects. There are no built-in named types, classes, or schemas in the language core. Complex types emerge by composing existing components or writing native ones, keeping the specification minimal and the type model fully open-ended.

Type and Data Constraints

Exon's property system turns plain fields into active validators. A property object defines get and set handlers that execute on every read and write. This lets you enforce type constraints, value ranges, format rules, and business logic at the field level, without scattering validation code through calling logic. The result is self-validating data structures with no external schema language required.

Use Cases

Because Exon has no fixed semantics, the same language adapts to a wide range of problems. Here are the most common ones:

  • Application Configuration: per-environment overrides via inheritance, secrets via process.env, no duplication across dev/staging/production.
  • Multi-format Document Generation: define a data structure once and emit JSON, YAML, XML, or plist by swapping a single wrapper component.
  • Scripting and Tooling: orchestrate build pipelines, run shell commands, generate GUI layouts, or drive any tool that consumes structured data.
  • GUI layout generator: describe a widget hierarchy as a structured object graph and compile it to target-language code, layout classes ready to be subclassed with event handler implementations.

Installation

The Exon CLI requires Node.js (v16 or later) and npm.

Quick Install

Clone the repository and run the install script:

git clone git@github.com:atdrez/exon.git
cd exon
bash install.sh

The script compiles the TypeScript runtime and installs the exon command globally via npm install -g. Once complete:

exon <file.exon>

Manual Install

You can also install manually from the TypeScript runtime directory:

cd runtimes/typescript
npm install -g .

Language

A high-level overview of Exon's design and syntax model.

File Structure

An Exon file contains exactly one top-level expression. That expression is typically one of three forms: 

// 1. Plain object
{
    key: value
}

// 2. Component call (e.g. encoding wrapper)
json.encode {
    indent: 4
    { key: value }
}

// 3. Inherited type with overrides
BaseType {
    override: "new value"
}

Nested Objects

{
    server: {
        host: "localhost"
        port: 8080
    }
    auth: {
        enabled: true
        provider: "oauth2"
    }
}

Inheritance

***
Extends Base.exon, inheriting the base properties and adding new ones
***

using models.Database  // load models/Database.exon

base {
    timeout: 30
    retryAttempts: 3

    database: Database {
        name: "development_db"
        poolSize: 20
    }

    // load models/Environment.exon
    environment: models.Environment {

    }

    collections: collections.Users {}
}

Multiline Strings

{
    message: """
    This spans
    multiple lines.
    """
  
    template: """{"key": "value"}"""
}

Comments

// single-line comment
port: 8080  // inline comment

***
multi-line block comment
spans as many lines as needed
***

Semicolons

Semicolons are optional separators. Both styles are equivalent:

{ host: "localhost"; port: 8080 }
{ host: "localhost"  port: 8080 }

Imports

using brings names into scope. Dot notation maps to file paths relative to the configured search paths. 

using fn.*             // all built-in components
using fn.string.join   // a single component
using models.Database  // import models/Database.exon
using lib.*            // all files under lib/

Component Calls

Any identifier followed by { } is a component call. Arguments inside the braces can be positional (values without a key) or named (key-value pairs). The component decides how to interpret them. 

using fn.*

{
    // positional args
    sum:  add { 10 5 }

    // named args
    text: string.join { "a" "b" "c"  separator: "-" }

    // nested calls
    url:  string.join { "http://" @root.host ":" @root.port }

    // lazy components receive unresolved args
    val:  if {
        condition: eq { @root.mode "dev" }
        then: "debug"
        else: "info"
    }
}

Anonymous Objects

An object without a key name is treated as the primary content of a component call. This pattern is used inside encoding wrappers: 

json.encode {
    indent: 4      // named arg for json.encode
    {              // anonymous object = the data to encode
        name: "app"
        version: 1
    }
}

The using fn.* Line

Most files begin with using fn.*. This brings all 90+ built-in components into scope under short names. Without it, use the full fn. prefix: 

fn.add { 10 5 }
fn.string.join { "a" "b" }
fn.process.env { "PORT" }

Selective imports reduce the available names and avoid collisions:

using fn.add
using fn.string.join
using fn.process.env

Inheritance

A file inherits from another by using that file's name as the root type. The resolver loads the referenced file, merges its properties, then applies the child's block on top. 

// Base.exon
{
    host: "localhost"
    port: 8080
    debug: false
}

// Child.exon  -- inherits from Base.exon
Base {
    timeout: 30
    retryAttempts: 3
}

Child.exon resolves to all properties from Base.exon plus timeout and retryAttempts.

Dot Notation and File Paths

Type nameResolved path
dbdb.exon
models.Databasemodels/Database.exon
infra.db.Prodinfra/db/Prod.exon

Chained Inheritance

// base/Server.exon
{ host: "localhost"  port: 8080  timeout: 30 }

// config/ApiServer.exon
using ..base.Server   // resolves to ../base/Server.exon

Server {
    port:    443
    timeout: 60
}

// app.exon
{
    server: config.ApiServer {
        host: "api.example.com"
    }
}

Partial Override with *

Use * instead of a full type name to merge only the listed properties into an existing nested object, rather than replacing the whole thing: 

Child {
    // replaces only name and poolSize inside the inherited database object
    database: * {
        name:     "mydb"
        poolSize: 50
    }
}

Or equivalently, using dot-path notation:

Child {
    database.name:     "mydb"
    database.poolSize: 50
}

Bindings

Bindings assign an identifier to an object so that other properties can reference it with @id.property. @root is always in scope.

@root

using fn.*

{
    port: 8080
    host: "localhost"

    baseUrl: string.join { "http://" @root.host ":" @root.port }
    server: {
        endpoint: string.join { @root.baseUrl "/api/v1" }
    }
}

Named Bindings

using fn.*

{
    info: Object@person {
        name:     "Jane"
        surname:  "Doe"
        fullName: string.join { @person.name " " @person.surname }
    }

    greeting: string.join { "Hello " @root.info.fullName }
}

Computed Bindings

using fn.*

{
    base:    100
    taxRate: 20

    tax:   mul { @root.base  div { @root.taxRate 100 } }
    total: add { @root.base @root.tax }
    label: string.join { "Total: $" @root.total }
}

Flow Control

All flow control components are deferred: they receive unresolved arguments and evaluate branches selectively.

if

{
    // result will receive object { x: 42 } or a string
    // depending if environment var ok is defined
    result: fn.if {
        condition: fn.defined { fn.process.env{"ok"} }
        then: { x: 42 }
        else: "nothing to see here"
    }
}

cond

{
    score: fn.coalesce { fn.process.env { "SCORE" } 0 }

    // grade based on score: A >= 90, B >= 80, C >= 70, else F
    grade: fn.cond {
        fn.ge { @root.score 90 }  "A"
        fn.ge { @root.score 80 }  "B"
        fn.ge { @root.score 70 }  "C"
        "F"
    }
}

switch

{
    result: fn.switch {
        value: fn.process.env { "CMD" }

        start:   "Starting service..."
        stop:    "Stopping service..."

        __default: fn.string.join { "Unknown command: " fn.process.env { "CMD" } }
    }
}

coalesce

{
    // use env port if valid or fallback to 8080
    port: fn.coalesce { fn.process.env{"PORT"} 8080 }

    // unlike or, treats 0 and false as valid (non-null) values
    retries: fn.coalesce { null null 3 }
}

foreach

{
    names: [
        { name: "Alice" surname: "Baggins" },
        { name: "Bob" surname: "Smith" },
        { name: "Carol" surname: "Potter" }
    ]

    // receive an array of greeting messages
    greetings: fn.foreach {
        data: @root.names
        do: fn.sequence@entry {
            person: fn.parameter{}

            // return greeting message
            fn.string.join {
                "Hello, " @entry.person.name " " @entry.person.surname "!"
            }
        }
    }
}

while

{
    target: 5
    counter: 0

    data: fn.while {
        // loop while counter < target
        condition: fn.lt { @root.counter @root.target }

        do: fn.sequence {
            fn.println { fn.string.join { "N=" @root.counter } }

            fn.set {
                target: @root 
                property: "counter"
                // increment counter by 1
                value: fn.add { @root.counter 1 }
            }

            // return counter
            @root.counter
        }
    }
}

try / raise

{
    port: fn.try {
        fn.sequence {
            // parse first argument as int
            fn.string.parseInt@value {
                fn.process.argv{1}
            }

            // if not number, raise exception (check fn.assert)
            fn.if {
                condition: fn.not { fn.number.is{ @value } }
                then: fn.raise{"PORT must be a number"}
            }

            // return parsed value
            @value
        }
        catch: 8080 // define 8080 if exception was raised
    }
}

Properties (getter/setter)

The property component defines active getter/setter pairs that validate or transform values on read and write. 

// lib/numclamp.exon
using fn.*

property {
    min: 0
    max: 100
    _value: 0

    get: get { target: @root; property: "_value"; }

    set: set { target: @root; property: "_value";
        value: sequence {
            assert {
                number.is { parameter{} }
                message: "Wrong assignment: must receive a number"
            }
            // clamp into min:max range
            math.clamp { parameter{} @root.min @root.max }
        }
    }
}
// constraints.exon
{
   number: lib.numclamp { _value: 30 } // initial value
}
{
   // a.number = 30 (inherited)
   a: constraints {}

   // b.number = 20 (overriden)
   b: constraints { number: 20 }

   // c.number = 0 (min clamped)
   c: constraints { number: -20 }

   // d.number = 100 (max clamped)
   d: constraints { number: 120 }

   // throw exception if uncommented (not a number)
   //e: constraints { number: "asd" }
}

Native Components

Load a JavaScript module as a component at runtime without modifying the Exon runtime:


// components/greet.js
module.exports.resolve = function(obj, context) {
    const name = obj.name || "World";
    return "Hello, " + name + "!";
};

// usage in app.exon
{
    fn.native {
        id:   "lib.greet"
        path: "components/greet.js"
    }

    message: lib.greet { name: "Alice" }
}

CLI

The Exon CLI is a single Node.js script at runtimes/typescript/bin/main.js.

Usage

node bin/main.js <file.exon> [options]

Options

OptionDescription
-e, --extendedPrint the parsed AST before resolution

Examples

# Basic run

# With environment variables
ENV=production PORT=443 node bin/main.js app.exon

# Extra search path
node bin/main.js app.exon -p ./shared -p ./lib

# Show parsed AST
node bin/main.js app.exon --extended

# Run test fixture
node bin/main.js ../../tests/fixtures/basic.test.exon -t

Other Editors

Exon does not yet have dedicated plugins for other editors. As a workaround, you can configure your editor to treat .exon files as a similar language (e.g. JSON or TypeScript) for basic highlighting. Contributions of grammar files for other editors are welcome.

Use Cases

Because Exon separates syntax from semantics, the same language adapts to a wide range of domains simply by swapping the component library. The examples below illustrate three common applications: scripting and build tooling, declarative GUI layout, and multi-format document generation.

Application Configuration

Define your application config once. Use inheritance for per-environment overrides and process.env for secrets that must not live in source control. 

{
    // if production, use api.example.com as host
    host: fn.if {
        condition: fn.eq { fn.process.env { "ENV" } "production" }
        then: "api.example.com"
        else: "localhost"
    }

    // use env PORT if valid, or fallback to 8080
    port: fn.coalesce { fn.process.env { "PORT" } 8080 }
}

Multi-format DocGen

Define your data structure once and produce JSON, YAML, PLIST, XML, or any custom format by using object wrappers. 

using fn.*

sequence {
    content: {
        name: "app"
        config: {
            host: "localhost"
            port: 8080
        }
    }

    // get first argument, or fallback to 'json'
    format: or { process.argv{1} "json"}

    cond {
        // output yaml
        eq {@root.format "yaml"}    yaml.encode { @root.content }
        // output xml
        eq {@root.format "xml"}     xml.encode { @root.content }
        // output plist
        eq {@root.format "plist"}   plist.encode { @root.content }
        // output json
        eq {@root.format "json"}    json.encode { @root.content }
    }
}

Scripting & Tooling

Exon can be used for tooling, configuration process, orchestrating build pipelines, generating gui layouts, and other tasks. 

*** Example of a build system ***

using fn.*
using lib.find_files
using lib.build_command

{
    assert {
        defined { process.argv{0} }
        message: "Script argument is missing"
    }

    dir: path.dirname { process.argv{0} }
    srcDir: string.join { @root.dir "/src" }
    outDir: string.join { @root.dir "/bin" }

    // create build command
    cmd: build_command {
        outputDir: @root.outDir
        includes: [ @root.srcDir ]
        target: or { process.argv{1} "hello" }
        files: find_files { dir: @root.srcDir path: "*.c" }
    }

    // make output directory & compile
    process.exec { "mkdir -p " @root.outDir ";" @root.cmd }
}

GUI layout generator

A Tk/Ttk widget hierarchy structured object graph that generates the corresponding Python code. Each widget type maps to a component; properties such as text, width, and command become named fields. The generated Python file wires up the widget tree and emits stub callback handlers for every bound event, ready to be filled in with application logic. 

using fn.*
using lib.ttk.*

Class {
    ref: "self"
    name: "TopFrameBase"
    parent_class: "ttk.Frame"

    // create label
    Label {
        ref: "label"
        text: "Your name:"
        layout: { anchor: "w" }
    }

    // create text input
    Entry {
        ref: "name_entry"
        owner: @root.ref
        layout: { fill: "x" pady: 5 }
    }

    // create button
    Button {
        ref: "btn"
        text: "Say Hello"
        layout: { pady: 5 }

        // bind click to method
        command: lib.py.lambda {
            target: @root.ref
            method: @root._say_hello.ref
        }
    }

    _say_hello: lib.py.def { ref: "_say_hello" }

    // methods to be overriden
    methods: [ @root._say_hello ]
}
